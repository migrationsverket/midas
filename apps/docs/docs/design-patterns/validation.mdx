# Validering i formulär

Validering ska hjälpa våra användare att fylla i formulär rätt. Syftet är att ge tydlig återkoppling vid fel och samtidigt underlätta för användaren att rätta till dessa.

## När och hur ska vi validera?

För att undvika att användaren blir överväldigad av felmeddelanden krävs att vi har en balans mellan att ge snabb återkoppling och låta användaren jobba ostört. Vissa fält ska ge omedelbar återkoppling när användaren lämnar dem, medan andra bara ska valideras vid efter att formuläret skickats.

### Fält som är obligatoriska men inte kan valideras mot format

Fält som användaren måste fylla i men inte har ett bestämt format eller regel valideras efter att användaren försökt skicka in formuläret. Det finns inget "fel" förrän fältet är tomt vid inskick så att validera dessa fält när användaren lämnar dem här riskerar att ge onödig stress och distraktion.

Exempel: Namn, fritextfält

<ValidateAfterSubmitExample />

### Fält med validerbart format eller regel

Fält där vi har ett bestämt format eller regel valideras när användaren lämnat fältet. Eftersom vi direkt kan avgöra om värdet är ogiltigt kan vi snabbt ge återkoppling, vilket sparar tid för
användaren och förhindrar fel längre fram. När användaren har rättat felet så återgår fältet till ursprungsläget och har inte längre något felmeddelande.

Exempel: e-postadress, telefonnummer, organisationsnummer/personnummer och postnummer

### Fält som kräver kontroll mot databas eller extern källa

Fält som kontrolleras mot databas eller extern källa valideras efter användaren lämnat fältet. [Visa en spinner om systemet behöver ladda.](./page-loading.mdx#spinner-bredvid-ett-fält)

Exempel: Kontrollera om kortnummer är giltigt

## Återkoppling

### Felmeddelande

Alla inmatningskomponenter har ett felmeddelande som kan visas vid valideringsfel. Felmeddelandet kan visas antingen ovanför eller under fältet. Som standard visas felmeddelandet ovanför fältet och det är detta vi rekommenderar eftersom det blir en logisk läsordning.

<ErrorMessagePositionExample />

### Riktlinjer för felmeddelandet

- Försök så långt det går att beskriva vad användaren ska göra för att det ska bli rätt. Undvik att skriva generiska felmeddelanden som ”fel format”.
- Undvik teknisk jargong och använd ett språk som användare förstår och känner igen.
- Använd en positiv och en icke dömande ton, undvik att skylla på eller antyda att användaren har gjort fel. Undvik till exempel ord som ogiltigt, otillåtet eller felaktigt.
- Använd samma ord i felmeddelandet som i övriga användargränssnittet.

## Felmeddelandelista

I tjänster som primärt är formulär som användaren ska fylla i, t.ex. e-ansökningar, använder vi förutom felmeddelande på varje kontroll även en Felmeddelandelista ovanför formuläret. Felmeddelandelistan är inte någon egen komponent utan består av en [Infobanner](/components/info-banner) med ankarlänkar till de komponenter som behöver rättas till.

Det viktiga är att `href` i listan matchar `id` på det fält det länkar till — det är så ankarlänken fungerar och hjälpmedel kan navigera direkt till fältet.

```tsx
const [name, setName] = useState('')
const [isInvalid, setIsInvalid] = useState(false)
const errorMessage = 'Du måste ange ett förnamn'

<Form onSubmit={e => {
  e.preventDefault()
  setIsInvalid(name.trim() === '')
}}>
  {isInvalid && (
    <InfoBanner type='warning' title='Justera dessa fält'>
      <ul>
        <li>
          {/* href måste matcha id på fältet nedan */}
          <Link href='#fornamn'>{errorMessage}</Link>
        </li>
      </ul>
    </InfoBanner>
  )}
  <TextField
    id='fornamn'
    label='Förnamn'
    isInvalid={isInvalid}
    errorMessage={errorMessage}
    value={name}
    onChange={value => {
      setName(value)
      if (value.trim() !== '') setIsInvalid(false)
    }}
  />
  <Button type='submit'>Skicka</Button>
</Form>
```
